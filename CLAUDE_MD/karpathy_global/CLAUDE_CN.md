# CLAUDE_CN.md

减少大语言模型（LLM）常见编码错误的规范指南。根据需要与项目特定的指令合并。

**折衷方案：** 这些指南更倾向于谨慎而非速度。对于简单琐碎的任务，请自行斟酌。

## 1. 动工前先思考

**不要盲目假设。不要隐瞒疑惑。把权衡方案提出来。**

在实现之前：
- 明确说出你的假设。如果不确定，请提问。
- 如果存在多种理解方式，请把它们都罗列出来——不要默默替用户做决定。
- 如果有更简单的方法，请说出来。在合理的场景下，学会拒绝或提出异议。
- 如果遇到不明确的地方，请停下来。指出让你困惑的点，并进行提问。

## 2. 简单至上

**用最少的代码解决问题。不做任何投机性的预测。**

- 不要提供超出需求的额外功能。
- 不要为仅使用一次的代码做抽象。
- 不要添加未被要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景编写错误处理。
- 如果你写了 200 行代码，而实际上 50 行就能解决，请重写。

问问自己：“资深工程师会觉得这太复杂了吗？”如果是，请简化。

## 3. 精准外科手术式修改

**只触碰必须修改的部分。只清理自己留下的痕迹。**

修改现有代码时：
- 不要“顺便改进”相邻的代码、注释或格式。
- 不要重构没有损坏或能正常运行的部分。
- 保持与现有代码风格一致，即使你的个人习惯并非如此。
- 如果你注意到了不相关的废弃代码（死代码），请提及它——但不要直接删除。

当你的修改产生孤立元素时：
- 删除那些因**你的**修改而不再使用的 import、变量或函数。
- 除非有明确要求，否则不要删除之前就存在的废弃代码。

检验标准：修改的每一行代码，都必须能直接追溯到用户的具体需求。

## 4. 目标导向执行

**明确成功标准。持续循环直到验证通过。**

将任务转化为可验证的目标：
- “添加校验” → “为非法输入编写测试，然后使其通过”
- “修复 Bug” → “编写一个能复现该 Bug 的测试，然后使其通过”
- “重构 X” → “确保测试在重构前后都能顺利通过”

对于多步骤任务，简要陈述一份计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰明确的成功标准能让你独立循环迭代。模糊的标准（如“让它跑起来”）则需要频繁的沟通澄清。

---

**如果达到以下效果，说明这些指南发挥了作用：** 代码差异（diff）中不必要的修改减少了、由于过度复杂而导致重写的次数减少了，并且澄清问题的提问出现在动工实现之前，而不是在犯错之后。
